### 0.2 The model

Three name fields, all stored in **Cyrillic Mongolian** as the canonical representation:

| Field | Required | Meaning |
| :--- | :--- | :--- |
| `ovog` | optional | Clan name (овог). Many members have none recorded. Never a sort key, never an identifier. |
| `etsgiin_ner` | required | Patronymic (эцгийн нэр) — the father's given name in the genitive. **Placed BEFORE the given name.** This is what occupies the surname slot on the passport/ID machine-readable zone. |
| `ner` | required | The given name (нэр). **This is the person's identity and the sort key.** All member-facing display, alphabetical ordering, search ranking, and salutation use `ner`. |

Plus two supporting fields and one derived field:

| Field | Required | Meaning |
| :--- | :--- | :--- |
| `mrz_name_latin` | required when an MRZ document is captured | The Latin name string **exactly as printed in the document MRZ**, stored verbatim. Never derived, never transliterated by the platform, never edited by the member. |
| `registration_number` | required | The 10-character national registration number (2 Cyrillic letters + 8 digits). The key for identity matching and vendor-result correlation. |
| `legal_name` | derived | **Retained term, redefined.** The read-only display composition `ovog` `etsgiin_ner` `ner` (or the document name as captured). Never independently stored or edited. See D9. |

Three normative rules the model must carry:

* **(a) Patronymic ≠ family name.** `etsgiin_ner` occupies the surname slot in the passport MRZ. Siblings therefore share a value; **a parent and child never do.** Any heuristic of the form "same surname ⇒ same household / same family / possible duplicate / related-party" is **wrong and must not be implemented** — in Mongolia it is false in both directions.
* **(b) Cyrillic is canonical; Latin is kept verbatim.** `ovog` / `etsgiin_ner` / `ner` are the system of record and are Cyrillic. Where a Latin form is needed (MRZ comparison, card embossing, international messaging), the platform uses `mrz_name_latin` **as captured**. The platform must never generate a Latin string by transliterating the Cyrillic fields.
* **(c) Match on registration number, never on name.** Duplicate detection, KYC vendor result correlation, sanctions/PEP screening correlation, state-register lookup, and record linkage key on `registration_number`. Name fields are **never** a matching key, neither alone nor in combination, and never as a fuzzy tiebreaker. (Addressing a P2P *payee* is a different operation and remains governed by DEC-3.)

**D2 — The two new fields are genuinely new.** `grep -rniE 'mrz|registration_number|registration number|latin'` over `final_requirements/` returns exactly two hits: `03:319` and `03:1256`. **Both match on the substring `latin` inside the word `violating`** — neither has anything to do with Latin script. Requirements (b) and (c) are therefore **not** achievable by renaming: without a verbatim Latin field there is nothing for embossing to use except a transliteration, and without a registration number there is no non-name matching key. Both must be introduced at `04:130` (entity), and threaded through `01:362`, `01:389`, `03:28`, `03:71`, `04:759`. `[VERIFIED: my own grep, re-run in rev 2]`

**D3 — The forbidden tokens cannot survive in any column.** `.softhouse/verify-docs.sh` computes `n_snake_name=$(c 'first_name|last_name')` via `grep -rEoh` over `idea-lab/final_requirements/*.md` and asserts it equals 0 under `hard`. `-o` counts every *occurrence*, not every line, and there is no context exclusion (unlike the money-type check, which does exclude `minor units|rejected|superseded`). So the tokens are prohibited even inside a "Superseded Variants" cell or a draft-comparison column. Both such places — DEC-6's own superseded column at `01:362`, and `04:109` — are paraphrased in the map. `[VERIFIED: .softhouse/verify-docs.sh, read directly in rev 2]`

**D4 — Card embossing needed a semantic fix, not a rename.** Seven occurrences say the embossed name is *derived from* the name fields (`02:219`, `03:610`, `03:649`, `04:315`, `04:829`, `04:1087`, and the theme summary `05:74`). Under a Cyrillic-canonical rule, "derived from the name fields" means the platform must invent a Latin transliteration — which is exactly what DEC-6(b) forbids. A pure field rename would preserve that defect in seven places. All are rewritten to *copy `mrz_name_latin` verbatim*, with an explicit block (`422 MRZ_NAME_UNAVAILABLE`) when it is absent rather than a fallback that guesses. `[VERIFIED: current text at those lines, read directly]`

**D5 — KYC correlation needed the same treatment.** `03:71` required OCR to populate the name fields before proceeding but said nothing about how the vendor result is tied back to the application. Renaming the fields there would satisfy the token check while leaving name-based correlation as the implicit default. The replacement makes `registration_number` the explicit correlation key, satisfying requirement (c) at the one place it actually binds. `[VERIFIED: current text at 03:71]`

**D6 — Rationale preserved; one clause retired as market-invalid.** DEC-6 argued that KYC vendors, card embossing and regulatory reports need decomposable name components. That argument is correct and is **retained in substance** in the replacement rationale, with an explicit statement that it produced this decision and that only the *shape* of the decomposition changes. The one clause that does not survive is "ACH/wire formats". It is retired **as market-invalid — that rail does not exist in Mongolia — not as reasoning-invalid**, and the replacement text says so in those terms. Rev 2 states this using the phrase "that rail" rather than naming the rail a second time, so the `rails` counter is unchanged (§0.3). `[VERIFIED: CLAUDE.md, "Payment rails — ACH / FedNow / SEPA / wire do not exist here"; counter arithmetic verified by my own grep]`

**D7 — Scope held to names.** DEC-6 also specifies a structured postal address. The address clauses are carried into the replacement row **verbatim and unchanged**. Mongolian administrative addressing (aimag → soum → bag; Ulaanbaatar düüreg → khoroo) and postal-code unreliability are a separate backlog item and are **not** addressed here. `[VERIFIED: task scope instruction]`

**D8 — Ordering is normative, not cosmetic.** The model states that `etsgiin_ner` precedes `ner` and that `ner` is the sort key. *Decision:* **option (a) — keep the token, redefine the meaning.** `legal_name` remains the name of the read-only derived display field, redefined as the composition `ovog` `etsgiin_ner` `ner` (or the document name as captured).

*Why:*
1. **Nothing forbids it.** The HARD check prohibits only `first_name|last_name`; `CLAUDE.md` and `patterns.md` say nothing about `legal_name`. Rev 1 retired a term the task never asked to retire.
2. **It is the smaller change.** Keeping it resolves all 10 occurrences with **zero** additional map items. Retiring it needs 4 new items and a decision on an API error code.
3. **It avoids a breaking API-contract change outside the task's scope.** `422 LEGAL_NAME_NOT_EDITABLE` is a published contract at `04:767`; renaming it is a client-visible break bought for no requirement.
4. **The redefinition is substantive, not cosmetic.** The composition rule changes (three Cyrillic fields in patronymic-first order, not first+last), and that change is stated in DEC-6, `01:389`, `03:28`, `04:109` and `04:130` — every place the term is *defined*. The places that merely *use* it (`03:165`, `03:169`, `04:766`, `04:767`) inherit the new meaning correctly with no edit. I checked each of the four and confirmed none of them asserts a first/last composition. `[VERIFIED: read 03:165, 03:169, 04:766, 04:767 directly]`

**D10 — `registration_number` precedence and capture channel. (F3)**

Rev 1 had the field arriving from two sources with no precedence rule. Resolved as follows, and written into DEC-6 as clause (d) plus items [8], [13], [15]:

* **Precedence:** an applicant-entered value is **provisional, for routing only**. The value returned by the **verified identity source** is authoritative. The stored value — and the `UQ` constraint — applies to the verified value.
* **Mismatch behaviour:** a disagreement between entered and verified values **blocks the application** (`409 REGISTRATION_NUMBER_MISMATCH`). Neither value silently overwrites the other. Guessing on the sole matching key is a compliance defect, not a UX inconvenience.
* **Card-face OCR is not a source.** `patterns.md` line 29: "**Post-2022 ID cards no longer print the registration number on the card face.** Any OCR flow that expects to read it there fails on current cards." Rev 1's item [8] required OCR to populate it — that requirement was unimplementable on current documents and is removed. `[VERIFIED: .softhouse/patterns.md line 29, read directly]`
* **What the verified source actually is remains undecided** — ХУР/XYP state-register lookup, MRZ, or an attested manual-entry flow. That is a blocker, not a silent assumption. See Blockers.

## Follow-ups

1. **Address model (explicitly out of scope here).** DEC-6 still carries a US/EU postal address (`city`, `region`, `postal_code`, `country`). Mongolian addressing is aimag → soum → bag, with düüreg → khoroo in Ulaanbaatar, and postal codes are unreliable. Separate backlog task; do not fold it into the application of this map.
2. **`03:177` mentions ACH inside the name-change scenario** — "the previous verified values remain in force for card embossing, ACH, and statements until the re-verification completes". No name token, so deliberately **not** in the map (payment rails are a different workstream). It sits three lines below item [9]; whoever fixes the rails should know it is there. `[VERIFIED: my own grep]`
3. **Decide genitive storage semantics.** Is `etsgiin_ner` stored already inflected (as printed on the ID), or stored as the father's nominative given name and inflected at render time? The replacement text is deliberately silent. Storing verbatim-as-printed is the safer default and matches "capture what the document says", but this needs a decision before schema work.
4. **Sort/collation rule.** `ner` is declared the sort key, but Mongolian Cyrillic collation (including Ө and Ү, which are not in Russian Cyrillic order) is not specified anywhere in the baseline. Needs a named collation before any member-directory or search story is implemented.
5. **Sweep for other prose-shaped occurrences.** Items [16] and [17] show that the token-based invariant check has a blind spot: the Western name model can persist in prose ("first/last name") while the check reports zero. Recommend a DRIFT counter for `first/last` and `given name/family name` phrasing. **Generalised in rev 2:** the same blind spot applies to *any* term a migration retires — the `legal_name` gap the reviewer found (F2) is the same class, discovered on the retiring side rather than the retired side. Any future map that stops using a term must census that term across all six documents before it is applied, precisely because no checker will notice.
6. **Confirm the two new error codes** (`422 MRZ_NAME_UNAVAILABLE`, `409 REGISTRATION_NUMBER_MISMATCH`) against whatever governs the API error vocabulary. They are proposals, not citations.
