# JOB 2 — Gate-dispute adjudication

I note the incentive structure before ruling. The orchestrator wrote `verify-docs.sh`, wrote the
rule in `SKILL.md` forbidding exceptions-to-clear-failures, and reaches PASS only under a
"checker defect" ruling. I have weighted that against the rulings below and have deliberately
declined the checker-defect route wherever the document could carry the fix instead.

I also examined the cited precedent rather than accepting it. The money-type HARD check's
`minor units|rejected|superseded` exclusion is **narrower than it looks**: it is scoped to a
structured migration table that enumerates rejected representations in a fixed vocabulary. It is
not a general licence to exempt prose that mentions a banned concept disapprovingly. Read as
precedent for "any sentence that disowns a term is exempt," it proves far more than it should.
I decline to extend it that far.

## DISPUTE 1 — `FDIC/NCUA` HARD check reports 2

### Ruling: **(b) Document defect.** Until the reword is ratified, **(c) — the gate stays red.**

### Finding of fact

Across the entire `idea-lab/final_requirements/` corpus there are **exactly two** hits on the
insurance HARD pattern, and **both are on line 29** — before the change and after it:

```
before:  ... an FDIC/NCUA-insured partner institution ...        -> FDIC, NCUA
after :  ... The previous FDIC/NCUA framing ... is withdrawn ... -> FDIC, NCUA
```

This is decisive, and it cuts against the checker-defect reading. This HARD check has exactly
one trigger in the whole baseline: the sponsor-bank row. It was written to drive that row to
zero. Narrowing the pattern so that a withdrawal-flavoured sentence is exempt would leave a
check that, by construction, now matches nothing anywhere — green not because the residue was
removed but because it was rephrased. That is the definition of a hollowed gate.

### Why not (a) checker defect

The semantic argument for (a) is real — a sentence that names FDIC/NCUA in order to withdraw
them does not *assume* US deposit insurance — but it fails on three counts:

1. **No pattern narrowing can be written safely here.** The exclusion would have to key on
   negation prose (`previous|withdrawn|does not apply|framing`). Those words are trivially
   co-locatable with a genuine claim ("FDIC coverage does not apply to share accounts, but
   deposits are insured up to..."), so the narrowed check would be defeatable by the exact
   class of sentence it must catch. Unlike the money-type case there is no structured construct
   to anchor the exclusion to.
2. **The tokens are not load-bearing.** `04` is a ratified requirements baseline, not a
   changelog. The withdrawal *history* belongs in the handoff record (`T6.md`, `T9.md`, this
   file) and in `05` OI-3, all of which already carry it. Nothing an implementer needs from
   line 29 requires the strings `FDIC` or `NCUA` to be present.
3. **The narrowing is being proposed by the party it unblocks.** Given (1) and (2) supply a
   zero-cost alternative, adopting the narrowing anyway would be loosening a gate purely because
   it blocked its author.

### Exact document fix (this is an amendment to text a product owner already approved)

Replace, inside the first column of line 29:

> The previous FDIC/NCUA framing assumed a US sponsor-bank/BaaS structure that does not apply in
> Mongolia and is withdrawn with no replacement claim about final form.

with:

> The previous US deposit-insurance framing assumed a sponsor-bank/BaaS structure that does not
> apply in Mongolia and is withdrawn with no replacement claim about final form.

Semantics are preserved exactly: same withdrawal, same "no replacement claim", same refusal to
decide the charter, still no assertion **or denial** of coverage (CLAUDE.md line 52 holds in both
directions). Only the two US-regime acronyms leave the baseline — which is precisely what the
HARD check exists to accomplish. Insurance HARD count: **2 → 0**.

This requires the product owner to re-ratify one clause. That cost is real and I am naming it
rather than routing around it. It is markedly smaller than the cost of the alternative.

## DISPUTE 2 — `rails` DRIFT rose 56 → 59

### Ruling: **split.** `ACH+` is **(a) a genuine checker defect** (1 of the 3). The remaining 2 are **(b) a document defect**. Until both are fixed, **(c) — the gate stays red.**

### 2a — `ACH+` matching `\bACH\b`: checker defect, fix required regardless of this change

`ACH+` is **Mongolia's** retail clearing system. `\b(ACH|FedNow|SEPA)\b` matches it because `+`
is a non-word character, so the trailing `\b` is satisfied. The check therefore scores a correct
Mongolian rail as a US-rail regression.

This is not narrowing-to-clear-a-failure. It is a bug on the migration's own success path: the
target state of `04` §4.3 and of this very column is text that *says* `ACH+`, per `CLAUDE.md`.
Every future correct rewrite will inflate the counter. Left unfixed, the DRIFT check penalises
the migration for succeeding. It must be fixed whether or not T9 merges.

**Exact pattern change** — in `.softhouse/verify-docs.sh`, replace:

```sh
n_rails=$(c '\b(ACH|FedNow|SEPA)\b')
```

with:

```sh
# ACH+ is Mongolia's clearing system, not US ACH. POSIX ERE is leftmost-longest, so the
# literal 'ACH\+' alternative wins over '\bACH\b' on "ACH+"; drop those matches.
n_rails=$(grep -rEoh 'ACH\+|\b(ACH|FedNow|SEPA)\b' "$DOCS"/*.md 2>/dev/null \
          | grep -vxF 'ACH+' | wc -l | tr -d ' ')
```

**What the narrowed check still catches:** every bare `ACH`, every `FedNow`, every `SEPA`,
anywhere in the corpus, in prose or comment, with no context exclusion whatsoever. The only
string that stops counting is the four-character token `ACH+`, which is a different system with
a different name. The gate loses no US-rail coverage at all. Effect here: 59 → 58.

### 2b — the two comment-internal `ACH` / `FedNow` repeats: document defect, **not** a checker defect

I explicitly **reject** the proposal to exclude matches inside `<!-- ... -->` comments. That
exclusion would create a permanent laundering channel: any US-market assumption could survive
the gate indefinitely by being wrapped in a comment. HTML comments are still read by
implementers, by diff reviewers, and by every agent that processes these documents — they are
part of the baseline in every sense that matters here. `.softhouse/uat.md` applies exactly as
written: the count went up, and a comment-exclusion would be re-baselining by another name.

The comment does not need to enumerate the US rails. It sits appended to the very column that
already names them; re-listing them is redundant. Replace:

> `<!-- TODO(rails): US rails. ACH, FedNow and wire do not exist in Mongolia; this column needs
> RTGS (Banksuljee), ACH+, and NETC card clearing. Separate backlog item — out of scope for the
> sponsor-bank correction. -->`

with:

> `<!-- TODO(rails): the US rails named in this column do not exist in Mongolia; it needs RTGS
> (Banksuljee), ACH+, and NETC card clearing. Separate backlog item — out of scope for the
> sponsor-bank correction. -->`

The backlog item keeps full specificity about the **replacement** rails, which is the part a
future implementer actually needs. Again an amendment to approved text; again small.
