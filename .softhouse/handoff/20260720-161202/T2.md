## Verdict: **REJECTED**

Two defects, one of which makes the project's own gate fail on application. The rails defect
alone would be a MICRO-FIX; the `legal_name` census gap is not mechanical (it needs a design
choice plus four new map entries), so the map must come back before it is applied.

The work is otherwise strong: the census of token-bearing lines is exactly right, the two prose
occurrences it claims the planning census missed are real and correctly identified, the semantic
embossing fix is the right call rather than a rename, and the provenance discipline is mostly

## Findings

### F1 — HIGH — Applying the map raises the `rails` DRIFT counter and fails `verify-docs.sh`

**Location:** T1 §1, the replacement text for `01_business_analysis.md:362`, rationale column.

**What is wrong.** `verify-docs.sh` computes
`n_rails=$(c '\b(ACH|FedNow|SEPA)\b')` using `grep -rEoh`, so it counts **every occurrence**,
not every line, and compares against `rails=56` in `.softhouse/baseline.txt`.

Line 362 today contains the literal `ACH` **once** ("ACH/wire formats"). The proposed
replacement contains it **twice**:

> The original rationale's "**ACH**/wire formats" clause is **retired as market-invalid** — **ACH** does not exist in Mongolia — not as reasoning-invalid

I confirmed the current count is exactly 56 and the replacement yields two matches. Applying the
map therefore produces `rails=57`, `+1 over baseline 56` → `FAIL`, and `VERDICT: FAIL`.

This is the precise failure mode the task's own D6 was trying to avoid: the author retired the
ACH clause correctly *in reasoning* but re-introduced the token *in text* while explaining the
retirement.

**Correct fix (mechanical, one line).** Remove the second literal token. Replace

`— ACH does not exist in Mongolia —`

with

`— that rail does not exist in Mongolia —`

Leaving one occurrence keeps the counter at 56 (`= baseline`, PASS). Removing both would give 55
and also PASS (a decrease is progress), so either is acceptable; do not add one.

**Note for whoever applies this:** `verify-docs.sh` on `main` today already reports
`FAIL US deposit-insurance regime (FDIC/NCUA) 2`. That is **pre-existing and unrelated** to
DEC-6 — do not attribute it to this change, and do not attempt to fix it here (scope).

### F2 — HIGH — Census gap: `legal_name` is orphaned in four places

**Location:** `03_acceptance_criteria.md:165`, `03_acceptance_criteria.md:169`,
`04_technical_architecture.md:766`, `04_technical_architecture.md:767`. None appear in the
migration map.

**What is wrong.** The map does not merely rename the three name fields — it also **retires the
term `legal_name`**, replacing "the derived `legal_name`" with "the derived display name" at
every location it touches (items [1]/[4]/[6]/[12]/[13], i.e. `01:362`, `01:389`, `02:77`,
`03:28`, `04:109`, `04:130`).

`legal_name` occurs at **nine** lines in the baseline. Five are inside map items. The other four
are not:

| Line | Current text | Status after applying the map |
| :--- | :--- | :--- |
| `03:165` | `* **And** the derived `legal_name` is displayed read-only with no edit affordance.` | references an undefined term |
| `03:169` | `* **When** a client submits a payload attempting to write `legal_name` directly, or submits a `postal_code` failing the country-specific format` | references an undefined term |
| `04:766` | `Profile per E-1 incl. derived `legal_name` (read-only), `membership_status`, ...` | references an undefined term |
| `04:767` | error column `` `422 LEGAL_NAME_NOT_EDITABLE` `` | **an API error code naming a field that no longer exists** |

This is exactly the failure the task brief warned about: a map applied mechanically leaves behind
what it failed to enumerate. It is invisible to `verify-docs.sh` — `legal_name` is not a checked
token, so the HARD check would report `first_name / last_name  0` and the run would go green with
four dangling references and a broken error-code contract still in the baseline.

The author's own Follow-up 5 identifies this blind-spot class ("the Western name model can persist
in prose while the check reports zero") but applies the insight only to `first/last` phrasing, not
to the term the map itself is retiring.

**Correct fix.** Pick one of two, state which, and make the map self-consistent:

- **(a) Keep the token.** Nothing in `CLAUDE.md`, `patterns.md` or `verify-docs.sh` forbids
  `legal_name`; only `first_name`/`last_name` are prohibited. Retain `legal_name` as the name of
  the read-only derived display field, redefined as the composition `ovog etsgiin_ner ner`. This
  is the smaller change: revert "display name" to "derived `legal_name`" in items [4], [6], [12],
  [13] and in §1, and the four orphans resolve themselves. `422 LEGAL_NAME_NOT_EDITABLE` stays
  valid.
- **(b) Complete the retirement.** Add four new map items with file, line, exact current text and
  exact replacement for `03:165`, `03:169`, `04:766`, `04:767`, including a decision on the error
  code (rename to `422 DERIVED_NAME_NOT_EDITABLE` and note it is a breaking API-contract change,
  or keep the old code string with a comment).

Option (a) is recommended — the rename buys nothing the task required, and it widens the blast
radius of a name-model change into the API error contract.

### F3 — MEDIUM — `registration_number` has two conflicting sources and no precedence rule

**Location:** map items [8] (`03:71`), [13] (`04:130`), [15] (`04:759`).

**What is wrong.** Item [15] adds `registration_number` to the **client-supplied**
`PATCH /api/v1/onboarding/applications/current` payload, and explicitly contrasts it with
`mrz_name_latin`, which it marks "KYC-populated, not client-supplied". Item [8] simultaneously
requires **OCR** to populate a non-empty `registration_number` before proceeding. Item [13] marks
the column `UQ`.

So the sole identity-matching key can arrive from two sources, and the map states no precedence,
no reconciliation rule, and no behaviour when the applicant-entered value and the OCR-extracted
value disagree — on the one field that duplicate detection, KYC correlation and screening
correlation all key on. A downstream implementer will guess, and guessing wrong on a matching key
is a compliance defect, not a cosmetic one.

Compounding it: `patterns.md` records that **post-2022 ID cards no longer print the registration
number on the card face**, so OCR of the card may not be able to supply it at all. The map does
not acknowledge this.

**Correct fix.** Add one normative sentence to the DEC-6 replacement, e.g.: *"`registration_number`
may be entered by the applicant for routing, but the KYC-verified value is authoritative; a
mismatch between the entered and verified value blocks the application rather than overwriting
either."* And add an `[UNVERIFIED]` note that post-2022 cards may not expose the number to OCR, so
the capture channel (ХУР/XYP lookup vs. MRZ vs. manual entry) is undecided.

### F4 — LOW — A `[VERIFIED]` tag whose own body admits half of it is unverified

**Location:** T1, Decision **D8**.

The tag reads
`[VERIFIED: CLAUDE.md non-negotiable — "Names are three fields..."; the patronymic-first ordering and sort-key rule come from the task brief, not from a repo source]`.

The disclosure is honest, but it is filed under `[VERIFIED]`, and a later reader scanning tags
will count it as sourced. The project has already lost work to fabricated confidence; the marker
must match the weakest claim it covers.

**Correct fix.** Split it: keep `[VERIFIED: CLAUDE.md ...]` for the three-field structure, and add
a separate `[UNVERIFIED]` bullet for patronymic-first ordering and for `ner`-as-sort-key. (Both
happen to be correct in fact and consistent with the ordering CLAUDE.md lists, but that is not
in-repo evidence.)

### F5 — LOW — D2's line attribution is wrong (its conclusion is not)

**Location:** T1, Decision **D2**.

D2 says the `mrz|registration_number|latin` grep "returns only two unrelated hits (`03:319`
'configurations', `03:1256` 'violating')". I re-ran it: both hits are the substring `latin`
inside the word **`violating`** — `03:319` matches on "configurations violating this are
rejected", not on "configurations". The substantive conclusion is **correct and I confirm it**:
neither `mrz_name_latin` nor `registration_number` exists anywhere in `final_requirements/`, so
both fields are genuinely new and (b)/(c) are not achievable by rename. Cosmetic only.

### F6 — LOW — "the sole identity-matching key" grazes DEC-3

**Location:** T1 §1, DEC-6 replacement text.

The replacement bolds `registration_number` as "**the sole identity-matching key**". DEC-3
(ratified, `01:359`) makes a P2P recipient addressable by `PHONE | EMAIL | MEMBER_ID`. These do
not actually conflict — DEC-3 governs *addressing a payee*, DEC-6(c) governs *identity matching
and record linkage* — and the DEC-6 row does go on to enumerate the bound cases correctly. But the
unqualified superlative sits two rows below DEC-3 in the same table and reads as an override.

**Correct fix.** Qualify: "the sole key for **identity matching, duplicate detection, KYC and
screening correlation, and record linkage** (P2P payee addressing is governed by DEC-3 and is
unaffected)."

### F7 — LOW — Inconsistent rule for prose occurrences, stated nowhere

**Location:** map items [12] vs [16]/[17].

Item [12] instructs the applier to **preserve** the prose "first/last, no address" in the
draft-comparison columns of `04:109`. Items [16] and [17] **rewrite** structurally identical prose
at `04:829` and `04:1087`. Both calls are individually defensible — 109 is a historical record of
what the superseded sprint drafts said, 829/1087 are live normative text — but the map never
states the distinguishing rule, and an applier working item-by-item will see a contradiction and
may "helpfully" normalise one to match the other.

**Correct fix.** Add one line above the map: *"Prose describing what a superseded draft said is a
historical record and is preserved verbatim; prose stating current platform behaviour is
rewritten. `04:109` is the only instance of the former."*
