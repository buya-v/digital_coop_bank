**FINDING 5 — MEDIUM / `05_prd_and_roadmap.md:159` (DEC-35) and
`03_acceptance_criteria.md:476` / live cross-document contradiction with the newly
ratified DEC-6. NOT a T4 defect — needs its own task.**

* `05:159` — `| DEC-35 | EP-4 | Recipient confirmation shows first name + last initial ("Member B.") | **CONFIRM** | …`
* `03:476` — `* **[PROPOSED]** Recipient confirmation shows display name as first name + last initial (e.g., "Member B.") to limit PII exposure (Sprint 1 draft).`

Under the model now ratified in `01` §6 there is **no last name**. `etsgiin_ner` is a
patronymic and `ner` is the identity, so "first name + last initial" would render a
Mongolian member as *given name + father's-given-name initial* — which inverts the
privacy property DEC-35's own rationale relies on ("full-name exposure enables identifier
fishing"): the initial would be the *less* identifying token and the fully-exposed one
would be the person's actual identity. The current text is not merely stale, it is wrong
in a way that produces a worse privacy outcome than intended.

**The applier was right not to edit them.** Doing so would have (a) improvised outside an
approved map, (b) amended a ratified decision other than DEC-6 — which this review's own
constraint forbids — and (c) silently made a privacy trade-off that DEC-35's rationale
shows was deliberated. Reporting it was the correct call, and the applier's Blockers
section states it clearly rather than burying it.

However: the baseline now carries a **DEC-6 vs DEC-35 contradiction**, which is precisely
this project's characteristic failure mode. It must not be left to a follow-up bullet.
**Correct fix: a new task to re-adjudicate DEC-35** (and the matching `[PROPOSED]` line at
`03:476`) for a three-part name model. The decision is non-obvious — plausibly
`ner` + `etsgiin_ner` initial, but that must be argued against the identifier-fishing
rationale, and it sits in DEC-3 payee-addressing territory that DEC-6(c) explicitly carves
out.
