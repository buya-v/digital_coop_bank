**The ratified decision (product owner, via AskUserQuestion — not re-opened here):** recipient
confirmation shows the standard Mongolian short form = **patronymic initial + given name**, e.g.
Cyrillic **"Ц. Бат"** (Latin "Ts. Bat"). Because the given name (`ner`, DEC-6) *is* the identity,
masking it is culturally wrong and would not preserve the anti-fishing property the way initialing a
family name does; so the anti-fishing defence **moves from name-masking to a per-sender velocity cap
on the recipient-lookup operation**, seeded in US-12.5.

## Decisions

**D1 — No new DEC is needed; DEC-35's amendment + one EP-4 rule line + the 04:810 note suffice.**
This is the smaller of the two options and I recommend it. Justification:
- DEC-35's amended *Rationale* already introduces the lookup-velocity cap as the relocated anti-fishing
  defence, so the normative decision is captured where the display-rule decision lives — the two are one
  and the same product decision (the PO ratified them together).
- The control is operationally a **US-12.5 configuration seed**, exactly like DEC-33's velocity seed —
  it is a config value, not a fresh normative decision that needs its own DEC number. A new DEC would
  duplicate DEC-33's seed pattern and re-state DEC-35's rationale.
- The enumeration protection it complements **already exists** (03:368 `403/404` convention; 04:810
  `404 RECIPIENT_NOT_FOUND` uniform response), so the map *references* it rather than re-creating it.
- Net footprint: 1 amended DEC row, 2 reworded 03 bullets, 1 new 03 bullet, 1 reworded 04 row. A new
  DEC would add a row and still require the same 03/04 edits. `[VERIFIED: DEC-33 at 05:157; 03:368;
  04:810; EP-4 rules block 03:468-480]`

**D2 — Census of the old display rule and of "Member B."-style display strings.** Two occurrences
*assert* the old "first name + last initial" rule: `05:159` (DEC-35) and `03:476` ([PROPOSED] line).
Exactly **one** occurrence *renders* it as a display string shown to a member: `03:486` ("the app shows
'Member B.'"). All other `Member [A-Z]\.` grep hits are actor prose or unrelated displays (enumerated in
"Lines deliberately NOT in the map"). The grep I ran:
`grep -rnE '"Member [A-Z]\.?"|DEC-35|first name|last initial|display name|recipient.confirm' *.md`
over all six `final_requirements/*.md`. `[VERIFIED: my own grep over idea-lab/final_requirements/*.md,
results read line-by-line]`

**D3 — The velocity cap carries no number, by design.** The new 03 bullet and the 04:810 `429` case both
state "recipient-lookups per sender per period" and "no hard-coded threshold / US-12.5 config seed / PO
to confirm." No integer is invented. This mirrors how DEC-33's own value is described as a "seed" that
"lives in US-12.5." `[VERIFIED: DEC-33 phrasing at 05:157 and 03:474]`

**D4 — 03:486 keeps the actor label to avoid breaking 03:10.** Rather than replace "Member B." with a
bare "Ц. Бат" (which would inject a placeholder personal name and contradict 03:10's "no placeholder
personal names"), the replacement shows *Member B's short-form display name … e.g. "Ц. Бат"*. The actor
stays "Member B"; the Cyrillic string illustrates the format. This is the least-invasive reading that
still lands the ratified display form. `[VERIFIED: 03:10 and 03:486 read directly]`
