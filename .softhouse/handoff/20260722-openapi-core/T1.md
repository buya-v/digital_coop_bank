## Changes Made

Replaced the placeholder `backend/openapi/root.yaml` (empty scaffold) with the real foundation:

- **`openapi: 3.1.0`**, **`info`** (title "Digital Coop Bank API", version 0.0.1, description noting derivation from `04` §3 and architecture-of-record status), **`servers: []`** (data residency is operator-chosen — no URL hardcoded).
- **`security`** document-level default: `memberOAuth2` (member OAuth2). Note in comments that webhooks / onboarding bootstrap override this at operation level in later slices.
- **`components.securitySchemes`** (4):
  - `memberOAuth2` — OAuth 2.0 Authorization Code + PKCE, RS256 JWT access tokens ≤15 min, OIDC ID tokens (§5.1). PKCE recorded via `x-pkce: S256` extension + prose (OpenAPI 3.1 has no native PKCE field). Auth URLs are placeholders (`*.example.invalid`).
  - `memberOIDC` — `openIdConnect` scheme for the OIDC ID tokens named in §5.1.
  - `staffSSO` — `openIdConnect`, separate staff token audience (§3.0 `/api/v1/admin/...`; §5.1 staff SSO + mandatory MFA). For later admin slices.
  - `stepUpAssertion` — `apiKey` header `X-Step-Up-Token` representing the fresh single-use MFA/biometric assertion required by §5.1 for sensitive actions. Augments (does not replace) `memberOAuth2`.
- **`components.responses`** (6 reusable): `Unauthorized` (401), `Forbidden` (403 FORBIDDEN / MEMBER_NOT_ACTIVE), `NotFound` (404), `Conflict` (409 STATE_CONFLICT / IDEMPOTENCY_CONFLICT), `UnprocessableEntity` (422 VALIDATION_FAILED), `TooManyRequests` (429 RATE_LIMITED). Each references `#/components/schemas/Error`.
- **`components.schemas.Error`** — the exact §3.0 envelope: `{ "error": { "code", "message", "details[]", "correlation_id" } }`. `error.code`, `error.message`, `error.correlation_id` required; `details` optional array.
- **`components.parameters`** — `Cursor` (`cursor` query string) and `Limit` (`limit` query integer) for cursor pagination (§3.0).
- **`tags`** — one per EP-1…EP-13 (names verbatim from §3.1–§3.13 headings) plus `ops` and `webhooks`, each with a one-line description.

## Decisions

- **Step-up represented as an `apiKey` header** (`X-Step-Up-Token`) rather than a scope or `x-` extension — it composes cleanly into operation-level `security` requirements alongside `memberOAuth2`, which the later sensitive-action slices need. `04` shows step-up minted by `POST /api/v1/auth/step-up` and, in §3.5, passed as a `step_up_token` body field; the header form is the cleanest reusable representation and is marked inferred.
- **Added `staffSSO` and `memberOIDC`** beyond the literal ask (member OAuth2 + step-up) because §3.0 references a separate staff audience and §5.1 names OIDC ID tokens; both are needed by later admin/auth slices and cost nothing here.
- **PKCE via `x-pkce` extension** — OpenAPI 3.1's `oauth2` object has no PKCE field; recorded as an extension plus prose so the requirement is not lost.
- **No `paths` key** in root.yaml — the validator injects `paths: {}` during assembly; keeping it out avoids a duplicate-merge and matches the task contract.
- **No currency / rail / vendor content** in the foundation — those are per-endpoint (§3.1+) concerns; the foundation stays neutral (relevant given the open MNT/vendor migration in CLAUDE.md, but out of scope for T1).

## Unverified (marked INFERRED in the spec)

- **Auth endpoint URLs** — `authorizationUrl`/`tokenUrl`/`refreshUrl`/`openIdConnectUrl` are `*.example.invalid` placeholders; concrete IdP endpoints are operator-chosen (§5.3 data residency), NOT pinned by `04`.
- **Step-up transport / header name** — `04` pins that a single-use step-up token exists and is returned by `POST /auth/step-up`, but not a uniform transport (§3.5 uses a body field). The `X-Step-Up-Token` apiKey-header representation and the header name are inferred.
- **PKCE encoding** — `x-pkce: S256` is an inferred convention; `04` says "with PKCE" without pinning the challenge method.
- **`Error.details[]` element shape** — `04` names the array but not the element schema; items left unconstrained.
- **`limit` bounds/default** — `04` names the `limit` param but pins neither default nor maximum; `1..100, default 20` is an inferred foundation convention.

## Follow-ups

- Later path slices must reference these tag names (EP-1…EP-13, ops, webhooks), the shared error responses (`#/components/responses/*`), the `Error` schema, and the `Cursor`/`Limit` parameters rather than redefining them.
- Sensitive-action operations (vote, external payment above threshold, card reveal, pledge, closure, proxy delegation, KYC-relevant profile change) should add `stepUpAssertion` to their operation-level `security` alongside `memberOAuth2`.
- Webhook and onboarding-bootstrap operations must override the document default with their own operation-level `security`.
- If/when the MNT + Mongolian-rail migration reaches the contract, currency/rail enums land in per-endpoint schemas, not this foundation.
