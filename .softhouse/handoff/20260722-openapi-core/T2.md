**Finding L1 — LOW / `components.securitySchemes.memberOAuth2` (`x-pkce: S256` + its INFERRED note).**
What: §5.1 says "with PKCE" and does **not** pin a challenge method. The handoff correctly lists "`x-pkce: S256` is an inferred convention" under Unverified, but the spec's own INFERRED note only explains *why PKCE lives in an `x-` extension* (OpenAPI 3.1 has no native field) — it does **not** state that the specific value `S256` is a chosen convention `04` does not pin. So a lone reader of the spec sees `x-pkce: S256` presented as fact. Every other inference in this file is flagged in-spec; this one value is the sole exception, and in-spec consistency of honesty is exactly what is under review here.
Severity rationale: LOW, not higher — `S256` is the RFC 7636-recommended method and a near-universal default, so it is a reasonable convention, and it *is* disclosed in the handoff. But it should be flagged in the spec too.
Fix: append to the `memberOAuth2` INFERRED note, e.g. "The challenge method `S256` is a chosen convention; `04` says 'with PKCE' without pinning a method."

No invented endpoint URL, header name, or enum is presented as fact. The `*.example.invalid` URLs are self-evidently placeholders and marked so. `X-Step-Up-Token` is marked inferred. No enums are invented (the foundation carries none beyond the security-scheme types). Clean on fabrication.

Minor, non-blocking notes (not defects):
- `staffSSO`/`memberOIDC` are typed `openIdConnect`; `04` says "staff SSO" / "OIDC ID tokens" without pinning that staff SSO specifically is OIDC. The `openIdConnectUrl`s are marked placeholder/inferred, and OIDC is a reasonable representation — acceptable, no in-spec claim that `04` pinned the protocol choice.
- `stepUpAssertion` as a reusable header scheme will not literally apply to `POST /cards/{id}/credentials` (which takes it in the body per §3.5); the spec discloses this. A later slice must not blindly attach the header scheme to that op.
