## Decisions

- **Shared enums defined once.** `MembershipStatus`, `KycStatus`, `Currency`, `DividendElection` live in `schemas/ep1-identity.yaml`; EP-2 schemas `$ref` them across the assembled doc (avoids the validator's duplicate-schema failure).
- **Security mapping.** Applicant/Member rows inherit the document default (`memberOAuth2`); the onboarding-application bootstrap overrides with `security: []` (04 Â§3.0 pre-auth bootstrap); step-up rows (`revokeDevice`, `updateMyProfile`, `createClosureRequest`) use `memberOAuth2` + `stepUpAssertion` as an AND requirement; the three `admin/...` rows use `staffSSO` (separate staff audience, 04 Â§3.0).
- **Money = `integer/int64`, "MNT minor units".** `par_value`, `amount`, `total_equity`, redemption `amount` all typed this way. Per the non-negotiable and the task rule, no floating point anywhere.
- **No config-seed constants hard-coded.** `04` shows the US-market example par/amount `2500` ($25.00); it is NOT pinned as a default/const/enum â€” par is US-12.5 config (DEC-11). Each such field's description says so explicitly.
- **`currency` is a free string, not an enum.** `04` Â§3.0/DEC-18 says the literal "USD"; the Mongolian baseline (CLAUDE.md) mandates MNT. Modelled as an ISO-4217 string code to avoid baking in the invalidated USD assumption â€” marked INFERRED representation. This is the one place I consciously did NOT reproduce a literal `04` value verbatim, because reproducing it would encode a known-invalidated non-negotiable violation; the description states the conflict openly.
- **Vendor field names kept, vendor identity flagged.** `persona_inquiry_id` and `payment_token` (Stripe) are named by `04` rows, so the fields exist; descriptions note the concrete vendor is market-dependent (CLAUDE.md flags Persona/Stripe as US assumptions).
- **HTTP success status codes are inferred** (201 for creates, 200 otherwise) â€” `04` names response bodies but never an HTTP status; each is marked INFERRED in the response description.
