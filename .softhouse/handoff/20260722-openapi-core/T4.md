## Decisions

- **Money non-negotiable.** All money/amount/balance fields use a shared `Money` schema (`type: integer, format: int64`, "MNT minor units"). No float anywhere. Its description records the honest tension: 04/DEC-18 fixes the literal `"USD"`; the project market is MNT — the re-denomination is a pending requirements change, not settled by 04. A separate `Currency` schema records 04's `"USD"` verbatim with the same pending-change note.
- **Idempotency-Key.** Defined once as reusable `IdempotencyKey` (header, required, uuid) in `schemas/ep4-payments.yaml` and `$ref`'d by every money-movement POST: EP-3 `savings-goals/{id}/transfers`, `group-pots/{id}/contributions`, `group-pots/{id}/outbound-requests`, `group-pots/outbound-requests/{id}/decision`; EP-4 `payments/p2p`, `payments/external`, `payments/schedules`, `payment-requests/{id}/settle`. See Assembly note below for why it lives under `components.schemas`.
- **Assembly constraint (why the parameter is a schema).** `validate.py` merges `schemas/*.yaml` into `components.schemas` ONLY and `paths/*.yaml` into `paths` ONLY; `components.parameters` can only come from `root.yaml`, which is approved and must not be edited. So `IdempotencyKey` — a valid Parameter Object — is stored under `components.schemas` and referenced as `$ref: '#/components/schemas/IdempotencyKey'` in `parameters` arrays. It validates as both a Schema Object and a Parameter Object (JSON Schema ignores the parameter keywords); the assembled doc reaches VERDICT: PASS.
- **External rails flagged, not renamed.** `payments/external` and `ScheduledPayment.rail` use 04's `ACH|WIRE|RTP` (+ `INTERNAL_P2P`) verbatim via `ExternalRail`/`Rail` enums, each carrying a PENDING REQUIREMENTS CHANGE note (Mongolia uses Banksuljee/ACH+/NETC, not ACH/FedNow). No Mongolian rail endpoint or enum invented; no rename.
- **Config values not hardcoded.** Velocity/step-up/lookup thresholds and fees are described as US-12.5 config only — no numeric limit is baked into any schema or enum. Step-up on `p2p`/`external`/`schedules` is conditional ("above config limit"/"for external") so it is documented in prose, not forced into operation `security`. The one unconditional step-up row — `group-pots/outbound-requests/{id}/decision` ("step-up") — carries `security: [memberOAuth2 + stepUpAssertion]`.
- **Common vs row-specific errors.** Rows' common codes (401/403/404/409/422/429) `$ref` the foundation's `#/components/responses/*`. Row-specific codes (402, 410, 502, and the named 4xx variants like `NOT_INTEREST_BEARING`, `GOAL_HAS_BALANCE`, `RECIPIENT_NOT_ACTIVE`, `LINK_IN_USE`, `PAYEE_IN_USE`, `EXECUTION_IN_FLIGHT`, `ALREADY_PAID`, `REMINDER_THROTTLED`, `RAIL_UNAVAILABLE`, `FEE_BALANCE_INSUFFICIENT`) are inline responses over the shared `#/components/schemas/Error` envelope, with the code named in the description.
- **Vendor leaks preserved with a note.** `plaid_public_token`, `plaid_item_ref`, and the `502 PLAID_UNAVAILABLE` code are recorded as 04 states them (verbatim), each with a note that the vendor is an assumption pending the Mongolia migration (CLAUDE.md). Not invented, not silently substituted.
- **Enums verbatim.** Every enum's values are copied exactly from §2.3/§2.4/§3.x. `LedgerEntryType` is the explicit union of all `TransactionType` values plus `HOLD_PLACED|HOLD_RELEASED|INTEREST_ACCRUAL` — derived from 04's "mirrors Transaction type + …" phrasing (INTEREST_POSTING already present in TransactionType, so not duplicated); no value invented.

## Unverified (marked INFERRED / type-inferred / pending in the files)

- **`Money` unit = MNT minor units** — 04/DEC-18 literally says `"USD"`; the MNT re-denomination is a pending requirements change per CLAUDE.md. Flagged in the `Money` and `Currency` schema descriptions.
- **`Idempotency-Key` on `group-pots/outbound-requests/{id}/decision`** — 04 §3.3 lists Idempotency-Key on the request-*creation* rows, NOT on this decision row. Applied here because threshold approval triggers a money movement and the project non-negotiable requires it. Flagged in the operation's requestBody description.
- **`Idempotency-Key` on `payments/schedules` POST** — 04 §3.4 does not list Idempotency-Key on this row. Applied per the project non-negotiable + T4 task (scheduled payments classified as money-movement POST). Flagged in the operation summary.
- **External-rail enums (`ACH|WIRE|RTP`, `INTERNAL_P2P`)** — verbatim from 04 but flagged as a pending requirements change (Mongolia rails).
- **Vendor references (`plaid_*`, `502 PLAID_UNAVAILABLE`)** — verbatim from 04, flagged as vendor assumption pending migration.
- **`type inferred; 04 does not specify`** on: `statement_delivery`, `progress_pct`, `AutoTransfer.frequency`, `ScheduleSpec.frequency`, `InterestHistory.postings[].period`, `expected_settlement` representation, `Payee.target_ref` (single string across 3 payee_type variants), `PaymentRequestShare.identifier` format.
- **INFERRED element shape** (04 names a JSON field but does not pin its schema): `Transaction.counterparty`, `GroupPotApprovalRequest.recipient` / outbound-request `recipient`, `micro_deposit_details`, `retry_policy`, group-pot `contribution_breakdown[].member` (rendered as member id).
- **`additionalProperties: false` NOT set** on the new entity schemas — 04's §2 attribute lists are read as the known fields, not a closed set (the foundation closes only the `Error` envelope, which §3.0 pins as "the exact shape"). Kept open to allow forward-compatible representation fields.

## Cross-slice coordination notes (for review / eventual merge with T3's EP-1/EP-2)
