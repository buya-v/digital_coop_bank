### 2. The `Idempotency-Key` parameter (T4) â€” NOT correct-as-is; needs restructuring.

T4 defines `IdempotencyKey` as a **Parameter Object** (`name/in/required/description/schema`)
and stores it under **`components.schemas`**, then `$ref`s it from operation `parameters:`
arrays (`$ref: '#/components/schemas/IdempotencyKey'`) on all EP-3/EP-4 money-movement POSTs.

This is **not valid OpenAPI 3.1 in the strict sense** â€” it passes only because the validator
is lenient:
- `components.schemas` is defined to hold **Schema Objects**. Interpreted as a schema, the
  object contains `name`/`in`/`schema` (non-schema keywords, ignored) and, critically,
  `required: true` â€” a **boolean where JSON Schema requires an array**. A strict
  schema-object check would reject it.
- A `parameters` array entry must be a Parameter Object or a `$ref` to a **Parameter
  Object**. `openapi_spec_validator` does not re-validate the `$ref` target against the
  Parameter Object schema at that position, so the category error slides through.

Stricter tooling / codegen will reject or mis-generate this. For an architecture-of-record it
is a genuine defect. **Recommended fix (cleanest):** add a `components.parameters` merge to
`backend/openapi/validate.py` (parallel to the existing `schemas/*` merge â€” ~6 lines), move
`IdempotencyKey` into a `parameters/*.yaml` slice (or a `components.parameters:` section), and
reference it as `#/components/parameters/IdempotencyKey`. `root.yaml` already holds
`components.parameters` (`Cursor`, `Limit`), so the container exists and the precedent is set;
the harness simply doesn't merge slice-level parameters today. **Fallback** (no tooling
change): inline the Parameter Object into each of the 10 operations' `parameters` arrays.
Note this defect is **isolated to T4** â€” T3's two idempotent POSTs use correctly-formed
**inline** Parameter Objects.
