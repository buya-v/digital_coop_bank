## Decisions

- **Scope = member rows only.** §3.7 has 6 admin rows, §3.9 has 2 admin rows,
  all prefixed "Admin —". Per the task (EP-12 admin `$ref`s my entity schemas)
  these are EP-12's to define. I own and define the entity schemas
  (`DividendDeclaration`, `DividendAllocation`, `CommunityGrantPool`,
  `SurplusMatch`) with the full E-32/E-33/E-45/E-46 enums so EP-12 can `$ref`
  them without redefining.
- **Idempotency-Key only where 04 lists it.** Only `POST .../backings` is a
  money movement (04 §3.9 lists Idempotency-Key on that row) → it `$ref`s the
  existing `#/components/parameters/IdempotencyKey`. Every governance /
  vote-submission POST is money-neutral → no key (per task rule). Dividend
  PAYOUT execution (the money movement) is on the EP-12 `/execute` row, not mine.
- **Secret ballot kept split.** `VoteParticipation` (who) and `VoteRecord`
  (what) are two separate schemas sharing only `ballot_id`; no schema, field, or
  endpoint joins a member to a choice. `submitVote` returns only a content-free
  `VoteReceipt`; `verifyVoteReceipt` returns valid/counted/superseded, never the
  choice (04 §5.1, E-39/E-40).
- **Reused shared schemas, never redefined:** `Money`, `Error`,
  `DividendElection` (E-1 enum, EP-1) for payout destinations,
  `RecipientIdentifierType` (EP-4) for the delegate identifier, and the shared
  error responses / `Cursor`/`Limit` params / `IdempotencyKey` param.
- **Step-up** expressed as `security: [{memberOAuth2, stepUpAssertion}]` on
  `submitVote`, `createProxyDelegation`, `revokeProxyDelegation` (04 §5.1).
- **Decimal(5,4) scores carried as strings** (`loan_repayment_performance_score`,
  `governance_participation_score`) to avoid a floating-point type — the money
  non-negotiable's spirit; 04 pins Decimal(5,4) but not a wire form.
