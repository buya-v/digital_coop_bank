## Entity-gating — the defining constraint of this task

`CLAUDE.md` flags EP-5 (Cards), EP-6 (Lending), and EP-10 (Round-Ups) as **unlawful for an SCC until the entity / payment-licence decision**. They are documented here for record completeness ONLY. Per the task's mandatory rule, **EVERY operation** carries:

```
x-entity-gated: true
x-entity-gated-reason: "Not buildable until the SCC entity / payment-licence decision (CLAUDE.md). Documented for completeness."
```

plus a one-line `description` note beginning "ENTITY-GATED (not buildable): documented for completeness only…".

**Confirmation: every gated operation carries `x-entity-gated`.** Verified mechanically — per file, the count of `x-entity-gated: true` equals the count of `operationId` +1, where the +1 is the file-header prose that names the tag (not an operation). No operation is missing the tag.

**Count of gated operations: 38** — EP-5: 10, EP-6: 25, EP-10: 3. (100% of the operations authored in T3 are gated; none is unmarked.)

## Decisions

- **Step-up (04 §5.1).** The `stepUpAssertion` scheme is required on the PAN / card-credential-reveal endpoints per the task rule: EP-5 `POST /cards/{id}/credentials` and `PUT /cards/{id}/pin` (PAN reveal + PIN set). `POST /cards/{id}/activate` also carries it — 04 §3.5 marks the row "step-up". EP-6 step-up rows that 04 §3.6 marks bold **step-up** also carry it: `offer-acceptance`, `pledges`, `signature-sessions`, `autopay`. Applied as `security: [{memberOAuth2: [], stepUpAssertion: []}]` (augments, not replaces, the session — matching root.yaml's scheme note). 7 operations carry it (EP-5: 3, EP-6: 4). The `credentials` row also keeps the 04 §3.5 body form `step_up_token` (root.yaml notes the transport is INFERRED).
- **Idempotency-Key (money-movement, CLAUDE.md non-negotiable).** `$ref '#/components/parameters/IdempotencyKey'` (the reusable parameter already defined in `params/ep4-payments.yaml`) applied to the two money-movement POSTs authored here: EP-6 `POST /loans/{id}/payments` (repayment — 04 §3.6 lists it explicitly) and `POST /loan-circles/{id}/pledges` (guarantee pledge — places a ledger hold; 04 does NOT list the key on this row, applied per the task rule + non-negotiable, flagged in-file). **No loan-disbursement and no round-up-transfer public endpoint exists** (both are internal engine behavior per 04 §3.6/§3.10), so those money-movements have no operation to carry the key.
- **BLOCKED loan rate model — NOT encoded (DEC-48/49).** No interest rate, APR, or basis-points field appears anywhere in the EP-6 slice. The 04 §2.6/§3.6 APR keys are intentionally **omitted**: `LoanProduct` drops `base_rate_apr_bps` + `circle_rate_discount_tiers`; the offer schema drops `standard_apr_bps` + `discounted_apr_bps`; `Loan` drops `apr_bps`. Installment / repayment / principal / interest / payoff amounts are modeled as typed `Money` only (`principal_due`, `interest_due`, `total_due`, `paid_amount`, `payoff_amount`, `interest_saved`, `accrued_interest`, `repayment_estimate`). No computed installment number (the blocked `$88.20`) is encoded. Each affected schema and operation carries an explicit "blocked pending DEC-48/49" note.
- **DEFERRED round-up denomination — NOT encoded.** The "nearest $1/$2/$5" denomination is deferred; no denomination value is encoded. `multiplier` (integer enum {1,2,3}) IS retained — it is the 04 E-47 field, not a denomination. `monthly_cap` is typed `Money` nullable; `batch_threshold` (04 E-47 config default "$5.00") is NOT in the §3.10 API surface and is not encoded.
- **Money non-negotiable.** Every amount/balance references the shared `Money` schema (`$ref '#/components/schemas/Money'`, integer minor units). No float anywhere. `Money`/`Currency` are NOT redefined — they are referenced from the existing shared slices.
- **Enums verbatim.** All enum values copied exactly from 04 §2.5/§2.6/§2.9 and DEC-20. `LoanStatus` (not previously defined) is created here per DEC-20 (`DRAFT|SUBMITTED|UNDER_REVIEW|APPROVED|ACTIVE|DELINQUENT|PAID_OFF|DEFAULTED|WRITTEN_OFF`) and used for both LoanApplication (origination subset, noted) and Loan (servicing). The loan schedule-type enum is named `LoanScheduleType` (`STANDARD|SEASONAL|INCOME_LINKED`) to avoid collision with the existing EP-4 `ScheduleType`.
- **Common vs row-specific errors.** Common codes (401/403/404/409/422) `$ref` the foundation `#/components/responses/*`. Row-specific codes are inline responses over the shared `#/components/schemas/Error` envelope with the code named in the description: EP-5 `STEP_UP_REQUIRED` (401), `CARD_NOT_ACTIVE`/`CARD_TERMINATED` (409), `ISSUER_UNAVAILABLE` (502), `ADDRESS_INCOMPLETE`/`MRZ_NAME_UNAVAILABLE`/`LAST4_MISMATCH`/`LIMIT_ABOVE_PRODUCT_MAX` (422); EP-6 `MEMBER_NOT_ACTIVE` (403), `NOT_DRAFT`/`CIRCLE_EXISTS`/`CIRCLE_ACTIVE`/`ALREADY_SIGNED`/`OPEN_HARDSHIP_EXISTS`/`IDEMPOTENCY_CONFLICT` (409), `OFFER_EXPIRED`/`INVITATION_EXPIRED` (410), `AMOUNT_OUT_OF_RANGE`/`INVITEE_NOT_ACTIVE`/`CIRCLE_FULL`/`POSITION_CONFLICT`/`EXCEEDS_AVAILABLE_BALANCE`/`SHARE_PLEDGE_UNAVAILABLE_IN_JURISDICTION`/`ACCOUNT_NOT_OWNED`/`EXCEEDS_PAYOFF` (422); EP-10 `PROJECT_NOT_PUBLISHED` (409), `DESTINATION_REQUIRED` (422). Every §3.5/§3.6/§3.10 "Key error cases" entry is represented.
- **Vendor leaks preserved with a note.** 04 names Lithic (card issuer). Recorded as an assumption pending the Mongolia migration (NETC is the Bank of Mongolia card entity, CLAUDE.md); the `502 ISSUER_UNAVAILABLE` code and PCI PAN-never-transits behavior are kept as 04 states. No Mongolian vendor invented, none silently substituted.

## TYPING — standard classes applied

- **uuid** (`type: string, format: uuid`): all ids/refs.
- **date-time** / **date**: timestamps (`created_at`, `sent_at`, `captured_at`, …) as date-time; `due_date`, `first_due_date`, `maturity_date`, payoff `as_of` as date.
- **Money**: every amount → `$ref '#/components/schemas/Money'` (shared).
- **enum-verbatim**: all status/type enums from 04 §2.5/§2.6/§2.9 + DEC-20.
- **inferred+description** (04 names the field but does not pin the shape): `Card.controls.per_period_limits[].period`, `delivery_address`, `affordability_inputs`, `decision.*` inner shape, `retry_policy`, `hardship_request`/`proposed_schedule.params`, `subject_refs`, `disclosure`, pooled-circle `ledger` view, `impact_summary`, `WalletProvisionResponse.status`, reveal-token `expires_at` representation. Each carries an `INFERRED`/`type inferred` note.

## Unverified (marked INFERRED / blocked / deferred / pending in the files)

- **Loan rate model / computed installment** — blocked (DEC-48/49); omitted from the contract, flagged in every affected schema + operation.
- **Round-up denomination** ("nearest $1/$2/$5") — deferred; not encoded, flagged in the EP-10 file headers + `updateRoundUpConfig` description.
- **Idempotency-Key on `/loan-circles/{id}/pledges`** — 04 §3.6 does not list it on this row; applied because the pledge posts a ledger hold (money-movement non-negotiable), flagged in the operation description.
- **Step-up on `/cards/{id}/activate`** — 04 §3.5 marks the row "step-up"; the `stepUpAssertion` scheme is applied (the task mandates it for PAN/credential reveal; activate is included as 04 marks it step-up).
- **`Money` unit = MNT minor units** — 04/DEC-18 literally says `"USD"`; the MNT re-denomination is a pending requirements change (inherited from the shared `Money` schema, not re-stated here).
- **Vendor reference (Lithic, `502 ISSUER_UNAVAILABLE`)** — verbatim from 04, flagged as vendor assumption pending migration.
- **`additionalProperties: false` NOT set** on the new entity schemas — consistent with the T4 convention (04 §2 attribute lists read as known fields, not a closed set; only the `Error` envelope is closed).
