## Decisions

- **Auth model — `security: []`, not member OAuth.** Every operation overrides the document-level `memberOAuth2` default with `security: []`. This is exactly what root.yaml §22-26 anticipates for webhooks and what §3.0 states ("Inbound vendor webhooks … signature-verified, no OAuth"). These are vendor-authenticated, not member/staff endpoints — so member OAuth2 is affirmatively removed, not merely omitted.
- **Signature verification represented as an `x-` extension, not a securityScheme.** Each operation carries `x-webhook-verification: '<verbatim §3.14 Verification column>'` plus a description note. Chosen over a new `securityScheme` because a scheme must live in `components.securitySchemes` (root.yaml, owned by foundation task T1) and the assembler merges only `paths/` + `schemas/` + `params/` from a slice — never `securitySchemes`. So a slice physically cannot contribute a security scheme without editing the approved foundation file, which is out of scope for T4. The `x-` note is the honest, in-scope representation; the task explicitly permitted "a security scheme **or** an `x-` note".
- **§4 four platform rules mapped concretely.** (1) signature-verified + replay-protected + idempotent → `x-webhook-verification` records the method; a replay/idempotency conflict surfaces as `409` (shared `Conflict`, which already names `IDEMPOTENCY_CONFLICT`); a replayed event returns the original `WebhookAck`. (2) correlation ID → `WebhookAck.correlation_id` echoed. (3) graceful degradation and (4) scoped rotatable keys are operational (not request/response shape), noted in prose only.
- **Lithic authorization is the one synchronous decision.** Its 2xx body is `LithicAuthorizationDecision` (`APPROVE`/`DECLINE`, optional `decline_reason`), not a bare ack — §3.14 says "Real-time auth decisioning (≤ 200 ms response)" and §4.4 confirms APPROVE/DECLINE within the 200 ms hard ceiling. It omits `409` (a synchronous decision is not an idempotent-replay surface the way async events are).
- **Money non-negotiable inside payloads.** `StripePaymentIntentWebhook.amount`, `LithicAuthorizationWebhook.amount`, `BaasStatusWebhook.amount` all `$ref` the shared `#/components/schemas/Money` (integer minor units). No float in any webhook field.
- **Vendor names verbatim + PENDING-change note.** Persona / Stripe / Lithic / Plaid appear verbatim in the paths per §3.14; each operation description flags the vendor set as a PENDING Mongolia requirements change (US vendors, not the Mongolian market — CLAUDE.md). The `esign` provider is unnamed in 04, noted as such. Vendor-leak field names recorded and cross-referenced: `inquiry_id` → `persona_inquiry_id`, `item_id` → `plaid_item_ref`.
- **BaaS rails not renamed.** `BaasStatusWebhook` records 04's ACH/wire/RTP status semantics verbatim with a PENDING-change note (Mongolia: Banksuljee (RTGS)/ACH+/NETC). No Mongolian rail invented.
- **Common vs row-specific errors.** `409`→ shared `Conflict`, `422`→ shared `UnprocessableEntity` (or an inline `422` where a specific code is named). Signature failure is an inline `401` over the shared `#/components/schemas/Error` envelope; the `esign` `422 HASH_MISMATCH` and Stripe `422 AMOUNT_MISMATCH` are inline over the same envelope.

## Unverified (marked INFERRED in the files)

- **Signature-scheme representation & header names.** 04 pins the verification *method* per row (HMAC, mTLS, JWT, hash check) but NOT concrete header names or signing mechanics — the `x-webhook-verification` values are verbatim §3.14; any wire-level detail is left unspecified and flagged INFERRED.
- **`401 SIGNATURE_INVALID` code string.** 04 §3.0's common codes do not name a signature-failure code; the string is INFERRED (envelope shape is the pinned `Error`).
- **All vendor payload shapes.** 04 does not pin vendor event schemas, so each body schema is an INFERRED envelope: documented purpose-bearing fields typed (`event_type`/`type`/`webhook_type`, ids, amounts, `return_code`, `envelope_hash`, `fulfilment_stage`), remainder open via `additionalProperties: true`. Every field 04 does not name carries an INFERRED note.
- **`WebhookAck` body shape** — 04 pins idempotency + correlation ID but not the ack body; `{status: RECEIVED, correlation_id}` is a chosen convention.
- **`LithicAuthorizationDecision` wire fields** — APPROVE/DECLINE semantics are pinned (§4.4); the field names are INFERRED.
