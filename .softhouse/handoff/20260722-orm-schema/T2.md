## PROMINENT FLAG FOR REVIEWER — E-31 score types (the one thing to review)

E-31 types `loan_repayment_performance_score` and `governance_participation_score` as **`Decimal(5,4)`** — 0..1 rate/score values, NOT money. The model-gate forbids `Numeric`/`Float` on every column (to make float-money mistakes structurally impossible), so these Decimals cannot be stored as `Numeric`.

**Decision:** stored as a **scaled integer** — value ×10_000 → integer in 0..10_000 (constant `SCORE_SCALE = 10_000` in `dividends.py`). This preserves the 4 decimal places of `Decimal(5,4)` **exactly**; the application divides by 10_000 to recover the score. `Integer` is permitted by the gate.

- Alternative considered: `String` (lossless but not SQL-comparable/aggregable) — rejected in favour of the exact integer form.
- NOT `Float`, NOT `Numeric` (both would fail the gate and violate the money non-negotiable's spirit).
- Reviewer decision needed: confirm scaled-integer ×10_000 is acceptable for these two score columns, or direct an alternative.

## Other Decisions

- **E-17 shares** modelled as a child table `PaymentRequestShare` (FK→`payment_request`) — the relational form of 04's nested "shares child rows". Its `debtor_identifier_type` value set (RecipientIdentifierType, DEC-3) is NOT enumerated in 04 §2.4 → `DeferredEnum` (String), not invented.
- **E-31 `period`** is a real single-value enum `PatronagePeriod.MONTH` (04 lists exactly one value). `period_key` string format (`"2026-01"`) is INFERRED (marked in-comment) — 04 names `period_key` in the UQ but not its format.
- **Cross-slice FKs modelled as bare UUID, not `ForeignKey`** (documented in-line) so this slice's model-gate stays self-contained (target tables absent from this worktree):
  - E-32 `factor_weights_config_ref` → ConfigurationParameter
  - E-47 `savings_goal_id` → SavingsGoal, `project_id` → CommunityProject
  Wire the real FKs at merge, once all slices are present. FKs to foundation tables (`member`, `account`, `transaction`, `membership_share`) and intra-slice (`payee`, `payment_request`, `dividend_declaration`) ARE real `ForeignKey`s.
- **JSON columns** (`schedule`, `retry_policy`, `fulfilment`, `controls`, `wallet_tokens`, `factor_weights`, `explainability`) use `JSONB` (postgresql dialect) per the rules.
- **`factor_weights` / split percentages NOT baked** — weights are configuration (FINANCIAL_POLICY ballots); the 10% Community Grant split is a ratified/config input, not a schema constant. `batch_threshold` (04's $5.00 default) is a Configuration Registry value (US-12.5), so no default amount is set on the column.
- **Ledger core untouched** — E-6/7/8 not modified. No config-seed values (par, limits) hardcoded.

## Honesty / market flags (mirroring 04 §2, per CLAUDE.md — flagged, not silently "corrected")
