# T3 — Lending ORM models (E-19…E-30)

Task: derive the EP-6 lending entities E-19 LoanProduct, E-20 LoanApplication,
E-21 LoanCircle, E-22 LoanCircleInvitation, E-23 PeerGuarantee, E-24 Loan,
E-25 RepaymentSchedule, E-26 RepaymentInstallment, E-27 PooledLoanCircle,
E-28 PooledLoanCircleParticipant, E-29 CollectionsCase, E-30 SignedDocument into
one file: `backend/app/models/lending.py`.

## What was done

- 12 tables, one per entity, `class X(Base, UUIDPrimaryKey, Timestamps)`, mirroring
  the membership.py / ledger.py pattern. Money -> `MoneyMinor`; ids UUID; date
  fields -> `Date`; JSON payloads -> `JSONB`; arrays -> `ARRAY(...)`.
- Enums are REAL only where 04 §2 lists the value set (16 real enums, all verbatim):
  `LoanStatus` (DEC-20, verbatim), LoanProductType, LoanProductStatus, ScheduleType,
  LoanCircleStatus, LoanCircleInvitationStatus, PledgeSource, PeerGuaranteeStatus,
  RepaymentScheduleStatus, RepaymentScheduleOrigin, RepaymentInstallmentStatus,
  PayoutOrderMode, PooledLoanCircleStatus, ParticipantPayoutStatus,
  CollectionsTrigger, CollectionsCaseStatus, SignedDocumentType, SignedDocumentStatus.
- Unique constraints modelled: E-21/E-24 loan_application_id UQ; E-22 UQ(circle,
  invitee); E-25 UQ(loan, version); E-26 UQ(schedule, sequence); E-28 UQ(circle,
  member) + UQ(circle, payout_position).

## §2 honesty ledger (04 §2 mirror — inferred / deferred / omitted, marked)

- **`LoanStatus` verbatim (DEC-20)** — DRAFT | SUBMITTED | UNDER_REVIEW | APPROVED |
  ACTIVE | DELINQUENT | PAID_OFF | DEFAULTED | WRITTEN_OFF. Used by E-20 and E-24.
  E-20's `DECLINED_CLOSED` is NOT added to the enum: DEC-20 does not list it and
  06 §8.8 flags it as an unauthorised addition; it is recorded in
  `LoanApplication.decision.outcome` (JSON), per the canonical decision.
- **E-20 `status` typed as the full LoanStatus enum** [INFERRED scope] — 04 says the
  application uses a DRAFT..APPROVED *subset*; the subset is a state-machine rule,
  not a distinct type, so the column carries the full glossary enum. Noted in code.
- **`addressed_via` (E-22) -> DeferredEnum (String)** [DEFERRED] — it is the glossary
  enum `RecipientIdentifierType` (PHONE|EMAIL|MEMBER_ID, DEC-3), owned by another
  model slice. Modelled as DeferredEnum here to avoid a cross-slice native-enum
  ownership/type-name collision; the owning slice defines the real enum.
- **Cross-domain references to tables NOT in this worktree modelled as plain UUID
  (NOT DB FKs)** [DEFERRED, so the gate resolves]: `config_version_ref`
  (-> ConfigurationParameter, E-19), `referral_case_id` (-> ComplianceCase, E-20),
  `approved_via` (-> MakerCheckerApproval, E-25), `assigned_staff_id`
  (-> StaffUser, E-29). These belong to other slices (T1/T2/T4/T5). Real
  `ForeignKey`s are used only for tables present here: member, account, transaction,
  ledger_entry (from membership.py/ledger.py) and the lending tables themselves.
  ACTION for integrator: promote these four columns to real FKs once the referenced
  tables land in the merged metadata.
- **`payment_transaction_ids` (E-26), `signer_member_ids` (E-30)** modelled as
  `ARRAY(UUID)` — arrays can't carry per-element FKs; they hold logical refs.

## Rate-model / APR block (DEC-48/49) — REQUIRED confirmation

**Zero interest-rate / APR / bps VALUES are encoded. Zero computed installments.**
- `LoanProduct.base_rate_apr_bps` (E-19) and `Loan.apr_bps` (E-24): plain nullable
  `Integer` placeholders, NO `default`, NO `server_default`, NO value written.
  Comment on each: "rate model BLOCKED (DEC-48/49) — column exists, no value/logic".
- **`circle_rate_discount_tiers` (E-19) OMITTED** — that JSON IS the DEC-49 discount
  curve, i.e. the blocked rate model itself. Omitted rather than stubbed; re-add
  only when the rate model is unblocked. (Only intentionally dropped 04 field.)
- `Loan.accrued_interest`, and E-26 `principal_due`/`interest_due`/`total_due`:
  these are AMOUNTS (MoneyMinor), stored by a servicing engine that does not exist;
  nothing here computes them. Modelling a money column is not encoding a rate.
- `LoanApplication.offer`/`decision` JSON carry apr_bps FIELDS at runtime only — the
  schema stores no rate value. Flagged in code.
- Verified by grep: no `default=` and no numeric rate literal (1200/800/600/400 etc.)
  anywhere in the file.

Also honoured: EP-6 is entity-gated / unlawful-until-the-entity-decision — this is
schema modelling for completeness only; no lending endpoint/flow/service built.
Ledger-core E-6/E-7/E-8 untouched.
