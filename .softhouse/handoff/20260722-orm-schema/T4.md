## Pattern conformance

- Every table is `class X(Base, UUIDPrimaryKey, Timestamps)` — with ONE deliberate exception (VoteRecord, see below).
- All money is `MoneyMinor` (BIGINT minor units), never float: Backing.amount; CommunityProject.funding_goal/amount_backed/amount_matched/amount_disbursed; CommunityGrantPool.budget/committed/disbursed; SurplusMatch.project_cap/accrued_amount/released_amount. That is 11 money columns in this slice.
- All ids are UUID; FKs are by table-name string (`ForeignKey("member.id")` etc.).
- Enums are real Enums ONLY where 04 lists the full value set; otherwise DeferredEnum / plain column. No enum value invented.

## SECRET-BALLOT INVARIANT (04 §5.1) — explicitly confirmed

**VoteRecord (E-40) has NO member linkage.** Verified structurally against the built metadata:
- `vote_record` columns: `ballot_id, choice, candidate_selections, validity_proof, voided, cast_at, id`. No `member_id`, no `member`-named column.
- `vote_record` foreign keys: exactly one — `ballot_id -> ballot`. No FK to `member` (direct or transitive).
- Columns shared between `vote_record` and `vote_participation`: only `ballot_id` (and each table's independent `id` PK, which is not a link). There is no relational path from a choice back to a member.
- `candidate_selections` is a bare `UUID[]` of ElectionCandidate ids — NOT a ForeignKey, and it references candidates, not members, so it opens no join to a voter.
- **VoteRecord deliberately does NOT inherit `Timestamps`.** A precise `created_at`/`updated_at` (server-default now()) would enable the timing correlation §5.1 forbids ("timestamps on choice records are coarsened"). Its only time field is `cast_at`, which the Governance Service coarsens. This mirrors ledger.py's LedgerEntry, which also omits Timestamps.
- VoteParticipation (E-39) is the ONLY governance table linking a member to a ballot; it carries no choice column. One-member-one-vote is expressed as UQ(ballot, member).

## Decisions / INFERRED (marked in-file)

- **Proposal.rejection_reason -> DeferredEnum (String), NOT a real enum.** 04 writes "`rejection_reason?` Enum **incl.** QUORUM_NOT_MET" — the value set is explicitly open, so per the ledger.py precedent it is a DeferredEnum, not an invented enum.
- **E-45 CommunityGrantPool.funded_from_declaration_id -> plain UUID, NOT a ForeignKey. [INFERRED cross-slice]** 04 says FK->DividendDeclaration (E-32), which lives in the Dividend Service slice (S-7) and is NOT defined in this worktree. A `ForeignKey("dividend_declaration.id")` would fail the standalone gate (unresolved FK). Modelled as a plain UUID column with a comment; the relational FK is to be wired at integration when both slices share one Base.metadata. Flagged for the merger.
- **Circular FKs** Ballot<->EligibilitySnapshot and Proposal<->Ballot are intentional (04 defines both directions). `Ballot.eligibility_snapshot_id` and `Proposal.ballot_id` are nullable (set at open/scheduling). SQLAlchemy resolves these by string; gate passes.
- **JSON fields -> JSONB** (context_pack, quorum_rule, results, auth_context, term, budget, timeline, documents, review, updates, impact_report, submitter_org, recurring). `UUID[]` fields -> `ARRAY(UUID)` with no FK (candidate_selections, release_transaction_ids).
- **Composite UQs declared** where 04 gives a deterministic pair: ProposalCosignature(proposal,member), EligibilitySnapshot(ballot), VoteParticipation(ballot,member), ElectionCandidate(ballot,member), SurplusMatch(pool,project), CommunityGrantPool(fiscal_year).
- **ProxyDelegation UQ(delegator, scope) WHERE status=ACTIVE NOT declared as a table constraint.** It is a PARTIAL unique index (predicate on status) best emitted by an Alembic migration; the "exactly one ACTIVE delegate per scope" and single-level rules are service logic. Noted in-file rather than approximated with a plain composite UQ that would be wrong.
- **ProxyDelegation.ballot_type_scope** modelled as a nullable BallotType enum with an in-file note that only BOARD_ELECTION is a valid scope value (constraint enforced in the service) — the enum values themselves are real (from 04), so no invention.
- **Derived money columns** (CommunityProject.amount_*, CommunityGrantPool.committed/disbursed, SurplusMatch.accrued/released) are typed columns only; their maintenance is the funding/ledger service's job, not the schema's — mirrors ledger.py's materialized balances. `match_ratio_bps` is a stored Integer (per task: fine — the ratio is data, not a baked constant).
