## Decisions

- **E-55 `value` is an OPEN type (JSONB), no baked seed.** This is the runtime config store (US-12.5): share par, P2P limits, round-up threshold, common-bond rules live here AT RUNTIME as JSON. The row structure is modelled (`key`, `value`, `version`, effective dating, approval FK); NO ₮ figure is a column default. Directly satisfies the task rule and mirrors membership.py's note that ₮10,000 par is config, not a schema default.
- **E-56 AuditLogEntry omits the Timestamps mixin — deliberate.** 04 makes it WORM (write-once-read-many, append-only). Like `LedgerEntry` (which also drops Timestamps for the same reason), an immutable record has no `updated_at`; its domain time is the explicit `occurred_at`. Documented in-file. This is a considered deviation from the `class X(Base, UUIDPrimaryKey, Timestamps)` default, justified by the WORM semantic + the LedgerEntry precedent. `prev_hash`/`entry_hash` are String columns only; chain computation and verification are the audit service's logic (not built).
- **E-53 checker ≠ maker is NOT encoded.** Both `maker_staff_id` and `checker_staff_id` (nullable) FK → `staff_user`; the inequality and the maker-checker-everywhere rule are constraint/logic concerns, modelled as the two columns plus an in-file note, per the task.
- **`Enum[]` fields → `ARRAY(Enum(...))`.** `default_channels` (E-49) and `roles` (E-52) are Postgres arrays of the real enum type — faithful to 04's `Enum[]` while keeping the values typed.
- **Nested status enums stay in JSON.** `Notification.channel_dispatches` lists its status set (QUEUED|SENT|...) INSIDE a JSON structure in 04, so it is JSONB, not a scalar enum column. Same for `ComplianceCase` sub-blobs.
- **E-59 money.** `total_managed_funds` is the one scalar money column → `MoneyMinor`. The per-category amounts in `allocations` live inside the JSON structure (04 models them nested), so they are integer minor units within JSON, not separate MoneyMinor columns; noted in-file. `source_ledger_hash` is a String column (reconciliation proof; proving it is logic).
- **FKs by table-name string, in-slice + to `member` only.** Resolving FKs: `notification`→`member`,`notification_event_type`; `notification_preference`→`member`; `maker_checker_approval`→`staff_user`(x2); `compliance_case`→`member`,`staff_user`; `configuration_parameter`→`maker_checker_approval`,`staff_user`; `data_subject_request`→`member`. All resolve against tables present in this worktree.

## Unverified / INFERRED (marked in-file)

- **All String column lengths are INFERRED.** 04 gives no lengths; sizes chosen conventionally (email 320, hashes 128, names/keys 128–256, notes 2000). Structural only.
- **E-55 `governing_ballot_id` is a bare UUID, NOT an enforced FK.** 04 says FK→Ballot, but the governance Ballot (E-33) table lives in another ORM slice absent from this worktree; a hard FK would not resolve and would fail the gate. Modelled as a plain `UUID` column with an in-file note for the merge to wire up. Its conditional mandatoriness (required for certified FINANCIAL_POLICY/GOVERNANCE_BYLAW outcomes) is logic, not a column.
- **`action_type` (E-53), `report_key` (E-58), config `key` (E-55)** are open Strings — 04 lists examples, not a closed value set. Not invented as enums.
- **`created_at`** on E-50/E-57 etc. comes from the Timestamps mixin (04 lists it); E-57's domain `received_at`/`resolved_at` and E-54's `opened_at`/`closed_at` are kept as explicit columns per 04 in addition to the mixin.
