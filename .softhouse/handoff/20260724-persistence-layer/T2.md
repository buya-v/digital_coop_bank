# T2 — wire /ready to a real DB liveness check

Branch: `softhouse/T2-ready-dbcheck`
Author role: Softhouse persistence-layer author (T2)
Scope: fulfil the promise already written in `main.py`'s `/ready` docstring,
using the T1 session layer (`app.db.session`, `app.api.deps.get_session`).
Sync only. `/health` untouched.

## Changes Made

- **`backend/app/main.py`**
  - New import list: `Depends`, `Response`, `status` from `fastapi`; `text`
    from `sqlalchemy`; `Session` from `sqlalchemy.orm`; `get_session` from
    `app.api.deps`; `engine_configured` from `app.db.session`; `Iterator` from
    `collections.abc`; `Optional` from `typing`.
  - New `ready_session()` — a local, `/ready`-only generator dependency (see
    Decisions).
  - Rewrote the `/ready` handler:
    - unconfigured → `200 {"status":"degraded","db":"unconfigured","currency":"MNT"}`
    - configured + `session.execute(text("SELECT 1"))` succeeds →
      `200 {"status":"ready","db":"ok","currency":"MNT"}`
    - any exception raised by that `execute()` call → caught, response status
      forced to `503`, body `{"status":"unready","db":"error","currency":"MNT"}`
      — the exception object/message is never read into the response.
  - Added return-type annotations `dict[str, str]` to both `health()` and
    `ready()` (they were bare `dict` before; this incidentally cleared two
    pre-existing mypy `[type-arg]` errors on `main.py` — see Tests).
  - `/health` body, route, and docstring: byte-for-byte unchanged.
- **New `backend/tests/test_ready.py`** — 3 tests, `TestClient` +
  `app.dependency_overrides[ready_session]`, no real DB touched (see Tests).
- **`backend/pyproject.toml`** — added
  ```toml
  [tool.ruff.lint.flake8-bugbear]
  extend-immutable-calls = ["fastapi.Depends"]
  ```
  This is the standard fix for ruff's `B008` ("no function calls in argument
  defaults") firing on FastAPI's own idiom, `param: T = Depends(...)`. This is
  the *first* route in the codebase to use `Depends`, so it's the first time
  this config gap showed up; every future feature route depending on
  `get_session` would have hit the same warning.

## Decisions

- **Configured/unconfigured dependency pattern: a local optional-session
  dependency, not a `try/except` around `Depends(get_session)`.**
  `app.api.deps.get_session` raises `DatabaseNotConfigured` *before* the route
  body ever runs when unconfigured (FastAPI resolves dependencies before
  calling the endpoint), so wrapping the endpoint body in try/except cannot
  intercept it — the exception surfaces as an unhandled 500 from FastAPI's
  dependency-resolution machinery, not a clean 200/503 from the handler.
  Instead I wrote `ready_session()`, a `/ready`-local generator dependency:
  ```python
  def ready_session() -> Iterator[Session | None]:
      if not engine_configured():
          yield None
          return
      yield from get_session()
  ```
  It checks `engine_configured()` itself (cheap, no I/O) and either yields
  `None` or delegates to the real `get_session()` generator via `yield from`
  (preserving `get_session`'s own `finally: session.close()`). The route then
  branches on `session is None`. This is the "check `engine_configured()`
  first, only touch the session in the configured branch" option from the
  task brief, expressed as one dependency rather than an inline check, because
  a single dependency is what makes the whole thing swappable in tests with
  one line: `app.dependency_overrides[ready_session] = <fake>`.
  **`ready_session` is deliberately not exported for reuse** — its docstring
  says explicitly that real feature routes should keep depending on
  `app.api.deps.get_session` directly, which stays strict (raises on a
  missing DB, exactly as T1 designed it). `/ready` is the one place
  "unconfigured" is a normal state instead of an error condition.
- **Broad `except Exception`, with `# noqa: BLE001` and an inline comment.**
  The task spec is explicit: "On ANY exception ... catch it, DO NOT leak the
  exception message/stacktrace into the response." Ruff's `flake8-blind-except`
  rule (`BLE001`) exists to stop exactly this pattern in general-purpose code,
  but a liveness probe's entire job is to convert any failure mode — driver
  error, closed pool, a leaked `DatabaseNotConfigured`, anything — into a
  clean 503 without ever re-raising or echoing it. I kept the broad catch and
  suppressed the rule locally with a comment explaining why, rather than
  weakening the catch to specific exception types (which would risk an
  uncaught error turning into a 500 with a real Python traceback — a worse
  leak than the one BLE001 guards against).
- **`Optional[Session]` instead of `Session | None` on the one
  runtime-evaluated annotation** — the `session` parameter of the `ready()`
  route handler. FastAPI/pydantic evaluates route-parameter annotations at
  import time to build the route's dependant signature. `backend/README.md`
  documents that the local dev machine here runs **Python 3.9** ("CI runs the
  full suite on Python 3.12"); `X | None` needs `type.__or__`, added in 3.10,
  so evaluating that annotation string on this host raises `TypeError:
  unsupported operand type(s) for |: 'type' and 'NoneType'` — [VERIFIED: hit
  this exact error live] confirmed by pytest collection failing with that
  traceback the first time I wrote it as `Session | None`. `Optional[Session]`
  works identically on 3.9 and 3.11+/3.12 (CI), so I used it there and added
  `# noqa: UP045` with a comment (ruff's `target-version = "py311"` prefers the
  new syntax, which is correct for the CI target but wrong for this host).
  Everywhere else the annotation isn't runtime-evaluated by FastAPI (e.g.
  `ready_session()`'s own return type `Iterator[Session | None]`), I kept the
  modern `X | None` spelling per the project's `py311` target — [VERIFIED: ran
  directly] confirmed both spellings coexist and the full suite still passes.
- **Response shape stability.** All three branches return exactly the same
  three keys — `status` (`degraded`/`ready`/`unready`), `db`
  (`unconfigured`/`ok`/`error`), and `currency` (always `CURRENCY_CODE`,
  "MNT") — matching the existing `test_ready_reports_mnt` assertion in
  `tests/test_health.py`, which I left untouched and re-verified passes
  unmodified against the new handler.
- **Bonus, not required:** typed `health()`'s and `ready()`'s return values as
  `dict[str, str]` instead of bare `dict`. This cleared `main.py`'s two
  pre-existing mypy `[type-arg]` errors as a side effect of touching the
  function signatures anyway; it wasn't requested but cost nothing extra and
  left `main.py` mypy-clean.

## Follow-ups

- If/when a real feature route needs a readiness-style "DB reachable" check
  reused outside `/ready`, consider promoting `ready_session`'s
  configured/unconfigured branching into `app.api.deps` as a named, exported
  dependency (e.g. `get_optional_session`) rather than copy-pasting the
  pattern. Left local to `main.py` for now since it currently has exactly one
  caller and T1's `deps.py` docstring is explicit that `get_session` itself
  should stay strict.
- `backend/pyproject.toml` now carries a `[tool.ruff.lint.flake8-bugbear]`
  section (previously the `[tool.ruff]` table had no `[tool.ruff.lint.*]`
  subsections at all). Every future route using `Depends(...)` benefits from
  this without needing its own `noqa`.
- The whole-project ruff run (166 errors) and the whole-project mypy run (8
  errors) both point at real, pre-existing debt outside this task's scope
  (`app/money.py` and model files' quoted-annotation style; `app/db/types.py`
  and model `dict` type-args; `app/db/check_models.py`'s `object.metadata`
  access). Not touched here — flagging so a future lint-debt pass has a
  starting inventory.
- This host's Python 3.9 vs. the project's declared `>=3.11` / CI's 3.12 is a
  recurring source of exactly this kind of `X | None` runtime-evaluation trap
  wherever FastAPI/pydantic needs to introspect a route signature. Worth a
  project-wide grep for other `Depends(...)`-adjacent `|`-union annotations
  before the next feature route lands, or getting a 3.11+ interpreter onto
  this dev machine so local verification stops being a special case.
