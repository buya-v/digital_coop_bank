# T3 ŌĆö offline CI gate: migration must match ORM metadata

Branch: `softhouse/T3-migration-gate`
Deliverable: an offline (no-database) CI gate that fails if `migrations/versions/0001_initial.py` drifts from `Base.metadata`.

## Changes Made

- **New `backend/app/db/check_migration.py`.** Offline drift checker, runnable as `python -m app.db.check_migration` (from `backend/`).
  - `render_model_ddl()` ŌĆö imports every model via `app.db.check_models._load_all_models()` (same technique `check_models.py`/`migrations/env.py` already use), then uses `sqlalchemy.create_mock_engine("postgresql://", dump)` + `Base.metadata.create_all(engine, checkfirst=False)` to compile the full DDL (`CREATE TYPE`, `CREATE TABLE`, `ALTER TABLE ... ADD CONSTRAINT` for `use_alter` FKs) without opening any connection. `create_mock_engine`'s URL is used only to select the dialect ŌĆö the "executor" callback (`_dump`) receives each compiled statement instead of it being sent anywhere.
  - `read_migration_ddl()` ŌĆö imports `migrations/versions/0001_initial.py` by file path (`importlib.util.spec_from_file_location`, since it lives outside any package on the default path) and reads its `_UP` list of hand-authored DDL strings.
  - `normalize_statement()` ŌĆö collapses incidental formatting only (see Decisions).
  - `compare()` ŌĆö normalises both statement lists into sets and returns `(matches, only_in_model, only_in_migration)`. This is the function the pytest test calls directly.
  - `main()` ŌĆö prints statement counts, and on mismatch prints the symmetric difference (`- ` = only in ORM metadata / missing from migration, `+ ` = only in migration / not derivable from current models), then `VERDICT: PASS`/`FAIL` and exits 0/1. Exit 2 on any exception (couldn't load models/migration at all ŌĆö the same convention `check_models.py` uses).
- **New `backend/tests/test_migration_matches_models.py`.** Calls `compare()`/`render_model_ddl()`/`read_migration_ddl()` directly (not the CLI process) ŌĆö `test_migration_matches_models` asserts the sets match with the diff embedded in the assertion message; `test_migration_ddl_is_nonempty` guards against a vacuous pass (both sides accidentally empty comparing equal).
- **`.github/workflows/ci.yml`** ŌĆö added `Migration matches models gate: python -m app.db.check_migration` in the `backend` job, immediately after the existing `ORM schema gate` step and **before** `Apply migration to Postgres` ŌĆö so drift is caught before any DB is touched, per the task requirement.

## Decisions (normalisation ŌĆö what was collapsed and why it's safe)

`normalize_statement()` collapses exactly three things, all incidental to a `str(Compiled)` rendering, never to SQL meaning:

1. Leading/trailing whitespace (`str.strip()`).
2. Every run of internal whitespace ŌĆö spaces, tabs, newlines ŌĆö down to a single space (`re.sub(r"\s+", " ", s)`). This is necessary because the mock engine's `str(sql.compile(...))` pretty-prints `CREATE TABLE` with a leading `\n`, tab-indented columns, and a trailing `\n\n`, while the migration's `_UP` strings were captured from the same technique but stored `.strip()`'d with the same internal tabs preserved as literal `\t` inside a triple-quoted string. Collapsing whitespace runs makes both renderings compare equal on structure without touching any token.
3. One trailing semicolon (`rstrip(";")`) ŌĆö the mock engine's DDL never emits one, but this makes the comparison robust regardless.

**Verified this is not too aggressive**: nothing else is touched ŌĆö keyword case, identifier names, column/constraint order *within* a statement, and every SQL token are left exactly as compiled. Proof: the bite-proof test below (┬¦Tests) added a genuine new column and the normaliser did **not** hide it ŌĆö the two `CREATE TABLE member (...)` strings (with vs without the added column) came out as two *distinct* normalised strings, `VERDICT: FAIL`. [VERIFIED: ran locally, see below]

**On current head the sets are exactly equal ŌĆö no drift found and no normalisation artifact needed beyond the above.** [VERIFIED: `python -m app.db.check_migration` ŌåÆ `models: 139 DDL statements ┬Ę migration: 139 DDL statements` ŌåÆ `139 statements match ŌĆö migration is in lockstep with the models` ŌåÆ `VERDICT: PASS`, exit 0.] This is expected per the task's robustness note: `0001_initial.py`'s own docstring says it was "Rendered offline from the ORM metadata" ŌĆö i.e. produced by this same mock-engine technique ŌĆö so a faithful comparison should pass, and it does.

Other decisions:
- `read_migration_ddl()` hardcodes the path to `migrations/versions/0001_initial.py` rather than discovering "the latest migration" generically. This matches the task's literal spec ("import `migrations/versions/0001_initial.py`") and the project's own stated constraint that the Alembic chain must stay linear (`migrations/env.py` docstring: "The chain MUST stay linear (ledger requirement)"). If/when a `0002_*` migration is added, this checker will need to compare against the migration chain's cumulative DDL rather than a single file ŌĆö flagged as a follow-up, not built here since only one migration exists.
- Raised `TypeError` (not `AttributeError`) when `_UP` is missing/not a list, per ruff's `TRY004` (this project's ruff config enables it ŌĆö see Blockers).
- `render_model_ddl()` imports `Base` directly from `app.db.base` rather than relying on `check_models._load_all_models()`'s return value, because that function is typed `-> object`; calling it only for its import side-effect (populating `Base.metadata`) and reading `Base.metadata` off the directly-imported, precisely-typed `Base` keeps `check_migration.py` itself mypy-clean without touching `check_models.py` (pre-existing `object`-typed return is out of scope for T3 ŌĆö see Blockers).
