## Decisions

- **Type name kept as `recipient_identifier_type`** (existing snake_case
  convention used by every other enum in this migration ‚Äî `kyc_document_type`,
  `device_platform`, etc.), not `recipientidentifiertype` as the task brief
  suggested. Renaming the Postgres type would touch the already-correct, already-
  created `group_pot_member.invited_via` column for no reason and break the
  project's established naming convention. [VERIFIED: grepped all ~85 `CREATE
  TYPE` lines in `0001_initial.py` ‚Äî every one is `snake_case`, none is
  concatenated.]
- **Owner is `deposits.py`, not `identity.py`** ‚Äî see premise correction above.
  `payments.py` and `lending.py` now both do
  `from app.models.deposits import RecipientIdentifierType`. Checked for import
  cycles: `deposits.py` imports only `app.db.base` / `app.db.types`; it does not
  import `payments` or `lending`, so no cycle. `check_models.py` loads modules via
  `pkgutil.iter_modules` in filename-alphabetical order (`deposits` sorts before
  `lending` and `payments`), so `deposits` is fully loaded before either
  referencing module even without the explicit import ‚Äî the explicit import is
  the correct/robust way regardless of load order. [VERIFIED: `grep -n "^from\|^import" app/models/*.py`.]
- **No divergent value set found** ‚Äî see semantic-equality section. Both
  candidate columns merged; nothing was left as `DeferredEnum`.
- Did not touch `ledger.py`'s `LedgerEntry.entry_type`/`Transaction.type` or
  `governance.py`'s `Proposal.rejection_reason` ‚Äî those `DeferredEnum` columns are
  unrelated concepts (ledger posting type; an open-ended rejection reason set),
  out of this task's scope, and a repo test
  (`tests/test_db_schema.py::test_ledger_core_defers_undecided_enums`) explicitly
  asserts the ledger ones stay `String`.
